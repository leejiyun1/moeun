# Workspace AGENTS Guide

## 목적

이 문서는 `/Users/ijiyun/moeun` 워크스페이스에서 작업하는 사람과 에이전트가 같은 기준으로 움직이기 위한 공통 운영 문서다. 이 워크스페이스는 프론트엔드와 백엔드가 분리된 멀티 리포 구조이므로, 기능 추가보다 먼저 계약과 책임 경계를 맞추는 것을 우선한다.

관련 설계 기준은 `docs/ADVANCEMENT_DESIGN.md`, `docs/BACKEND_ARCHITECTURE.md` 를 함께 본다.

## 사용자 기준

이 워크스페이스에서의 기본 의사결정 기준은 아래와 같다.

- 완성도 우선. 빨리 끝나는 구현보다 오래 유지되고 확장 가능한 구현을 택한다.
- 리팩터링은 제한하지 않는다. 더 좋은 방향이 명확하면 기존 구조를 언제든 개선한다.
- 불필요한 호출은 최대한 줄인다. 네트워크 요청, 중복 쿼리, 불필요한 상태 동기화는 부채로 본다.
- 일관성을 강하게 유지한다. 같은 문제는 같은 방식으로 푼다.
- 하드코딩을 최소화한다. 경로, 정책, 메시지, 상수, 분기 기준은 중앙화 가능한지 먼저 검토한다.
- 레거시처럼 보이는 임시 패턴을 늘리지 않는다. 프레임워크의 정석과 팀 규칙에 맞는 깔끔한 코드를 우선한다.
- 역할별 책임을 확실히 나눈다. 한 계층이 다른 계층의 일을 대신하지 않게 한다.
- 동작만 확인하고 끝내지 않는다. 운영 관점, 장애 포인트, 이후 고도화 여지까지 함께 본다.

## 저장소 구성

- `ExcellentFE`: React 19 + Vite + TypeScript 기반 프론트엔드
- `EasyBE`: Django + DRF 기반 백엔드

두 저장소는 별도 Git 저장소지만, 실제 서비스 관점에서는 하나의 제품으로 취급한다. API 계약, 인증 흐름, 추천/검색 경험은 항상 양쪽을 함께 보고 수정한다.

## 공통 작업 원칙

1. 문서 먼저
- 구조 변경, API 정책 변경, 인증 흐름 변경, 추천 로직 변경 전에는 설계 문서를 먼저 갱신한다.
- 구현이 설계를 앞서면 설계를 즉시 따라잡는다.

2. 계약 우선
- 백엔드 엔드포인트, 응답 형식, 인증 방식이 바뀌면 프론트 타입/상수/호출부를 같은 변경 단위에서 맞춘다.
- 프론트에서 백엔드 미구현을 우회하기 위해 하드코딩 URL이나 임시 분기 로직을 늘리지 않는다.

3. 점진적 개선
- 전면 재작성보다 레이어 정리, 중복 제거, 경계 명확화, 테스트 추가를 우선한다.
- 기존 구조를 무시한 새 패턴 도입은 금지한다. 필요하면 이행 단계와 제거 계획을 같이 남긴다.
- 일단 동작하는 해법으로 멈추지 않는다. 더 나은 구조가 명확하면 같은 작업 안에서 고도화 방향까지 반영한다.
- 프레임워크 관용구를 거스르는 구현보다, 정석적인 구조를 우선 채택한다.

4. 중앙화
- 인증, 토큰 갱신, 공통 에러 처리, API base path, 환경 변수 접근은 중앙 모듈에서 처리한다.
- 화면 컴포넌트와 개별 API 함수에서 같은 예외 처리 로직을 반복하지 않는다.
- 하드코딩 값은 상수, 설정, 정책 모듈로 올릴 수 있는지 먼저 검토한다.

5. 검증 가능한 변경
- 사용자 여정에 영향이 큰 변경은 최소한 `lint`, `build`, 관련 테스트 또는 수동 검증 절차를 남긴다.
- 테스트를 아직 못 붙이는 변경이라면 왜 못 붙였는지와 보완 계획을 기록한다.

6. 운영 관점 기본 적용
- 요청 수, 쿼리 수, 캐시 가능성, 실패 복구 방식, 로그 추적성을 같이 본다.
- 기능 하나를 만들 때도 운영 중 반복될 비용이 크면 초기에 구조를 바로잡는다.

## 아키텍처 경계

### Frontend

- `pages`: 라우트 단위 화면 조합
- `components`: 재사용 UI와 화면 조각
- `hooks`: 화면 또는 도메인 로직
- `api`: HTTP 요청/응답 변환
- `stores`: 전역 상태, 세션 상태
- `constants`, `types`, `utils`: 순수 보조 계층

프론트에서는 아래 규칙을 지킨다.

- `api` 계층은 브라우저 UI 부수효과를 직접 만들지 않는다.
- `pages`는 가능한 한 얇게 유지하고, 데이터 조합은 `hooks`로 내린다.
- 전역 상태는 인증, 사용자, 장바구니처럼 여러 화면이 공유하는 정보에만 사용한다.
- 서버 상태는 React Query를 우선 고려하고, 전역 store에 중복 저장하지 않는다.
- 같은 도메인의 호출 방식, 에러 처리 방식, 쿼리 키 규칙은 일관되게 맞춘다.
- 화면 구현 시에도 불필요한 재요청과 중복 계산을 먼저 의심한다.

### Backend

- `views`: 요청 진입점, 권한, 응답 조립
- `serializers`: 입력 검증, 출력 직렬화
- `services`: 비즈니스 로직
- `models`: 영속성, 도메인 상태

백엔드에서는 아래 규칙을 지킨다.

- Django `views`는 최대한 컨트롤러 역할만 담당한다.
- `views`에 쿼리 조립과 도메인 계산 로직을 넣지 않는다.
- `serializers`는 검증과 데이터 변환에 집중하고, 추천/주문 같은 핵심 로직을 떠안지 않는다.
- 재사용되는 조회/계산 로직은 `services`로 올린다.
- API 경로는 버전 정책을 따르며, 새 엔드포인트는 원칙적으로 `/api/v1/` 하위에 둔다.
- 쿼리는 기본적으로 중복 조회, N+1, 불필요한 annotate/ordering 가능성을 같이 검토한다.
- 임시 예외 분기나 문자열 하드코딩으로 정책을 숨기지 않는다.
- 모델, 서비스, 시리얼라이저, 뷰의 책임이 섞이면 우선 분리한다.

## 우선 기술 부채

현재 워크스페이스에서 먼저 다뤄야 할 부채는 다음과 같다.

- 백엔드 API prefix가 `/api/` 와 `/api/v1/` 로 혼재함
- 프론트 인증 초기화와 401 처리 책임이 `App`, store, axios에 나뉘어 있음
- 프론트 테스트 러너와 검증 기준이 사실상 비어 있음
- 로컬 실행 기준과 환경 변수 문서가 저장소별로 충분히 고정되지 않음

새 기능은 위 부채를 악화시키지 않는 범위에서만 추가한다.

## 저장소별 기본 명령

### ExcellentFE

- 설치: `npm ci`
- 개발 서버: `npm run dev`
- 린트: `npm run lint`
- 빌드: `npm run build`

주의:

- 테스트 러너는 아직 기본 세팅이 없다. 테스트 체계를 도입하는 변경은 설계 문서에도 남긴다.
- 환경 변수는 `VITE_` prefix를 사용한다.

### EasyBE

- 의존성 설치 기준: `uv` 또는 가상환경 + `requirements.txt`
- 마이그레이션: `uv run python manage.py migrate`
- 개발 서버: `uv run python manage.py runserver`
- 타입 검사: `uv run mypy .`
- 포맷 검사: `uv run isort . --check --diff`, `uv run black . --check`
- 테스트: `uv run python manage.py test`

주의:

- 설정은 `envs/.local.env` 와 Django settings를 기준으로 동작한다.
- 로컬 실행 시 DB/Redis 환경 변수가 필요하다.

## 변경 체크리스트

아래 항목 중 해당되는 것은 반드시 확인한다.

- API 경로와 프론트 상수가 같이 수정되었는가
- 인증/토큰 관련 변경이 중앙 처리 흐름을 깨지 않는가
- 중복 로직이 새로 생기지 않았는가
- 불필요한 호출이나 조회가 늘어나지 않았는가
- 하드코딩 대신 재사용 가능한 기준으로 올릴 수 있는가
- 설계 문서와 실제 구현이 어긋나지 않는가
- 검증 방법이 남아 있는가

## 완료 기준

작업은 아래를 만족할 때 끝난 것으로 본다.

- 구현 범위가 문서에 정의된 범위와 일치한다
- FE/BE 계약 불일치가 없다
- 최소 검증 결과를 설명할 수 있다
- 후속 작업이 필요하면 TODO가 아니라 명시적인 다음 단계로 정리되어 있다

## 문서 동기화 규칙

아래 변경이 발생하면 설계 문서를 같이 갱신한다.

- API 버전 정책 변경
- 인증 흐름 변경
- 검색/추천 로직 변경
- 저장소 구조 변경
- 운영/배포 방식 변경
