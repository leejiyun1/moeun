# Moeun Platform Advancement Design v1

## 문서 목적

이 문서는 `ExcellentFE` 와 `EasyBE` 를 하나의 제품으로 보고, 다음 고도화 작업의 기준과 우선순위를 정의하기 위한 설계 문서다. 목표는 기능을 더 많이 만드는 것이 아니라, 구조 부채를 줄이면서 추천 커머스 제품으로서 완성도를 높이는 것이다.

## 대상 범위

- 사용자 인증과 세션 유지 흐름
- 취향 테스트, 검색, 추천, 상품 상세, 후기, 장바구니, 주문 흐름
- 프론트엔드와 백엔드 사이의 API 계약
- 로컬 개발, 검증, 운영 안정성에 필요한 기반 작업

## 범위 제외

이번 문서 기준에서 아래는 직접 범위에 넣지 않는다.

- 결제 시스템 신규 도입
- 관리자 기능 전면 재설계
- 인프라 공급자 변경
- 전면 재작성 수준의 프레임워크 교체

## 현재 상태 요약

### Frontend

- React 19, Vite, TypeScript, React Router, React Query, Zustand 사용
- `pages`, `components`, `hooks`, `api`, `stores` 구조는 존재함
- 인증 초기화와 401 대응 책임이 여러 계층에 분산되어 있음
- 테스트 러너와 자동화된 UI 검증 기준이 사실상 없음

### Backend

- Django + DRF 기반 앱 분리 구조 사용
- `users`, `products`, `taste_test`, `feedback`, `cart`, `orders` 등 도메인이 나뉘어 있음
- 서비스 계층이 일부 존재하지만, API 정책과 URL 버전 체계가 완전히 통일되지 않음
- CI에서 타입 검사, 포맷 검사, 테스트를 일부 수행하고 있음

## 핵심 문제

1. 계약 불일치 가능성
- 프론트는 `/api/` 경로를 전제로 한 상수를 사용하고, 백엔드는 `/api/` 와 `/api/v1/` 가 혼재한다.

2. 인증 책임 분산
- 토큰 저장, 인증 초기화, 401 처리, 화면 이동이 여러 계층으로 흩어져 있다.

3. 품질 게이트 부족
- 프론트는 테스트 체계가 약하고, 백엔드는 실행 환경 의존성이 강하다.

4. 추천/검색 고도화 여지
- 검색은 필터 중심이고, 추천은 향후 피드백/행동 데이터 기반 정교화 여지가 크다.

5. 운영 문서 부족
- 로컬 부트스트랩, 필수 환경 변수, 변경 원칙이 루트 기준으로 정리되어 있지 않다.
- 루트 Docker 실행 기준은 `docker-compose.local.yml` 과 `docs/DOCKER_LOCAL_ENVIRONMENT.md` 로 고정한다.

## 고도화 목표

### 제품 목표

- 사용자가 로그인, 취향 테스트, 탐색, 장바구니, 주문, 후기 작성까지 끊김 없이 이동할 수 있어야 한다.
- 추천과 검색이 단순 나열이 아니라 사용자 취향과 행동을 반영해야 한다.
- 마이페이지에서 취향 데이터와 주문/후기 데이터가 하나의 경험으로 연결되어야 한다.

### 기술 목표

- FE/BE 계약을 명확히 하고 버전 정책을 통일한다.
- 인증/에러 처리/환경 변수 접근을 중앙화한다.
- 고위험 도메인 로직에 대해 테스트 가능한 구조를 만든다.
- 로컬 실행과 배포 기준을 문서화한다.
- 호출 수와 상태 흐름을 단순화해 운영 비용을 줄인다.
- 하드코딩을 줄이고 정책/상수/구성 기준을 중앙화한다.
- 프레임워크 정석에 맞는 구조로 정리해 레거시성 높은 패턴을 줄인다.
- 계층별 책임을 분명히 해 유지보수성을 높인다.

## 비타협 기준

아래 기준은 이번 고도화 동안 지켜야 하는 비타협 규칙이다.

1. 새 API는 원칙적으로 `/api/v1/` 아래에 둔다.
2. 인증과 401 처리 로직은 중앙 모듈에서 관리한다.
3. 프론트는 하드코딩 URL과 임시 alert 중심 에러 처리를 늘리지 않는다.
4. 백엔드 핵심 로직은 `view` 가 아니라 `service` 중심으로 모은다.
5. 기능 추가와 동시에 최소 검증 기준을 남긴다.
6. 불필요한 네트워크 호출과 중복 쿼리를 구조 부채로 본다.
7. 더 나은 방향이 명확하면 리팩터링을 주저하지 않는다.
8. Django view는 컨트롤러 역할에 집중시키고, 비즈니스 로직은 별도 계층으로 분리한다.

## 설계 원칙

### 1. Evolutionary Architecture

현재 구조를 존중하면서 점진적으로 정리한다. 전체 재작성은 금지하고, 다음 릴리스에도 안전하게 실을 수 있는 단위로 쪼개서 진행한다.

### 2. Single Source of Truth

- API 경로 정책은 백엔드 URL 구조와 프론트 상수에서 하나의 기준을 따른다.
- 인증 상태는 중앙 세션 흐름을 기준으로 해석한다.
- 취향/추천 계산 규칙은 백엔드 서비스 계층을 기준으로 둔다.
- 메시지, 상수, 정책성 분기, 환경 의존 값은 가능한 중앙 기준으로 모은다.

### 3. Thin Entry, Rich Domain

- 프론트 `pages` 와 백엔드 `views` 는 가능한 한 얇게 유지한다.
- 조합과 계산은 각각 `hooks`, `services` 로 이동한다.
- 프레임워크가 기대하는 책임 경계를 우선 따르고, 예외는 명시적으로만 허용한다.

### 4. Observable Change

성능, 오류, 품질 변화를 추적할 수 있어야 한다. 최소한 빌드, 린트, 테스트, 핵심 사용자 흐름 검증이 기준선으로 있어야 한다.

## 목표 아키텍처

### 1. API 계약

#### 방향

- 백엔드 엔드포인트를 단계적으로 `/api/v1/` 기준으로 수렴시킨다.
- 기존 `/api/` 경로는 필요 시 호환 레이어를 잠시 유지하되, 신규 구현은 허용하지 않는다.
- 에러 응답 형식과 인증 실패 처리 기준을 명시한다.

#### 적용 원칙

- 프론트 `constants/apiPaths.ts` 는 백엔드 URL 정책과 동시에 수정한다.
- 리스트, 상세, 생성/수정/삭제 엔드포인트 이름 규칙을 통일한다.
- 계약 변경 시 프론트 타입 정의와 API 호출부를 같은 작업 단위에서 맞춘다.

### 2. Frontend 구조

#### 목표

- 라우팅 계층은 화면 조합만 담당
- 데이터 패칭과 비즈니스 조합은 `hooks`
- HTTP 통신은 `api`
- 인증/사용자/장바구니 같은 공유 상태만 `stores`

#### 구체 작업

- 인증 초기화 흐름을 별도 세션 모듈로 정리
- 401 처리와 토큰 재발급 부수효과 정리
- 화면별 데이터 접근 패턴을 React Query 중심으로 재정리
- 검색/추천/마이페이지 등 주요 페이지에 로딩, 에러, 빈 상태 기준 추가
- 쿼리 키, 캐시 정책, 재요청 정책에 일관된 규칙 부여
- 상수와 정책 분기를 페이지 내부 하드코딩 대신 공통 모듈로 이동

### 3. Backend 구조

#### 목표

- `views` 는 요청 흐름 제어와 권한 처리에 집중
- `serializers` 는 입력 검증과 응답 포맷에 집중
- 핵심 계산은 `services` 로 이동
- 쿼리 최적화와 캐시 적용 지점을 서비스 단위에서 관리
- Django 관점에서 읽었을 때 정석적인 구조로 보이도록 유지

#### 구체 작업

- URL 버전 정책 정리
- 검색 필터 로직과 추천 로직의 서비스 경계 정리
- 후기, 주문, 취향 테스트 간 데이터 연결 구조 점검
- N+1 및 중복 쿼리 방지 기준 정리
- 운영 중 자주 바뀌는 정책 값은 상수 또는 설정 계층으로 이동
- view / serializer / service / model 책임이 섞인 구간 우선 정리

### 4. 추천/검색 고도화 방향

#### 검색

- 현재 필터링 기반 검색을 유지하되, 정렬과 랭킹 기준을 명시한다.
- 향후 검색어, 필터 조합, 클릭 데이터를 분석 가능한 형태로 남긴다.

#### 추천

추천은 아래 입력값을 점진적으로 결합한다.

- 취향 테스트 결과
- 시음 후기와 태그
- 조회/클릭/좋아요/구매 이력
- 재구매 또는 반응 패턴

1차 목표는 규칙 기반 추천 품질 개선이다.
2차 목표는 점수화 모델 도입이다.

### 5. 품질과 운영

#### Frontend

- `lint`, `build` 를 기본 게이트로 유지
- 테스트 러너를 도입해 핵심 훅과 API 경계를 검증
- 환경 변수 예시와 로컬 실행 문서 보강
- Docker 로컬 실행 기준은 루트 compose를 기준으로 유지
- 핵심 페이지에서 요청 수와 재렌더링 비용을 점검

#### Backend

- 로컬 개발 기준 환경 변수 문서화
- local/dev/prod DB 접속 정보 분리 원칙 문서화
- 테스트 실행 절차와 의존 서비스(DB/Redis) 명시
- 타입 검사와 포맷 검사를 PR 기준선으로 유지
- 주요 조회 API에 대해 쿼리 최적화 포인트를 문서화

## 단계별 실행 계획

### Phase 0. 기반 정리

목표:

- 문서 기준 수립
- API 정책 통일 방향 확정
- 인증 흐름과 환경 변수 기준 고정

작업:

- 루트 `AGENTS.md`, 설계 문서 정리
- API prefix 정책 정리
- 프론트 인증/세션 흐름 분해 계획 수립
- 로컬 실행 체크리스트 정리
- 공통 상수화/중앙화 우선 대상 식별

완료 기준:

- 팀이 같은 기준으로 작업할 수 있다.
- 신규 작업이 더 이상 구조 부채를 악화시키지 않는다.

### Phase 1. 사용자 흐름 안정화

목표:

- 로그인, 취향 테스트, 검색, 상세, 장바구니, 주문, 후기 흐름 안정화

작업:

- 인증/401 처리 중앙화
- 검색 필터 및 상세 데이터 흐름 정리
- 주문/후기 연계 흐름 점검
- 에러 상태와 빈 상태 UI 기준 수립
- 불필요한 호출과 중복 상태 제거

완료 기준:

- 핵심 사용자 여정에서 끊기는 화면/계약 오류가 크게 줄어든다.

### Phase 2. 추천 품질 향상

목표:

- 취향 기반 추천의 설명 가능성과 일관성 확보

작업:

- 취향 테스트와 후기 데이터를 묶는 추천 점수 기준 설계
- 개인화 리뷰/추천 API 기준 정리
- 추천 결과 검증용 샘플 시나리오 작성
- 추천 계산의 확장 포인트와 운영 튜닝 포인트 정의

완료 기준:

- 추천 결과가 단순 랜덤/정적 노출이 아니라는 것을 설명할 수 있다.

### Phase 3. 성능과 운영 고도화

목표:

- 응답 속도, 배포 안정성, 문제 추적성 향상

작업:

- 캐시 적용 후보 정리
- 로그/에러 추적 기준 강화
- 느린 API와 불필요한 프론트 재요청 식별
- 하드코딩된 정책과 운영 상수를 설정 계층으로 이동

완료 기준:

- 병목 지점을 수치로 설명할 수 있고, 개선 우선순위가 명확하다.

## 측정 지표

초기 베이스라인은 추후 측정으로 채우되, 최소한 아래 항목은 관리한다.

- 검색 API 응답 시간
- 추천 API 응답 시간
- 주요 화면별 요청 수
- 프론트 메인/검색/상세 페이지 빌드 크기와 초기 로딩 체감
- 인증 실패율 및 토큰 재발급 실패율
- 핵심 사용자 여정별 오류 건수
- lint/build/test 통과율

## Definition of Done

고도화 작업은 아래를 만족해야 완료로 본다.

1. 설계 문서 기준과 구현 결과가 충돌하지 않는다.
2. FE/BE 계약이 같은 작업 단위 안에서 맞춰져 있다.
3. 변경 이유와 검증 방법을 설명할 수 있다.
4. 이후 단계로 넘길 리스크와 남은 작업이 분리되어 있다.

## 오픈 이슈

- `/api/` 와 `/api/v1/` 호환 정책을 얼마나 오래 유지할지
- 프론트 테스트 러너를 Vitest 중심으로 둘지, 브라우저 기반 검증까지 같이 도입할지
- 추천 품질 평가를 어떤 운영 지표로 볼지
- 성인 인증과 소셜 로그인 흐름을 어느 수준까지 공통 세션 레이어로 묶을지

## 다음 단계 제안

문서 기준이 잡혔으므로 다음 작업은 아래 순서가 적절하다.

1. API 버전 정책과 프론트 상수 체계 정리
2. 인증/세션 흐름 리팩터링 설계
3. 프론트 테스트 베이스 도입
4. 검색/추천 서비스 경계 재정의
